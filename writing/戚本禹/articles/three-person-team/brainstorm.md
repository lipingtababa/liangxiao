## 理论基础：从上一篇文章延续

上一篇文章的核心观点：

> "把AI当成手下刚入职的实习生。你会发现，那些让你头疼的技术缺陷，都会迎刃而解。"

**四个关键洞察**：

1. **AI不可靠是因为问题本身复杂**：你作为manager要"吸收那些它处理不了的复杂性"
2. **幻觉问题**：用"知人善任"和"情境领导力"来管理——信任是渐变光谱，不是非黑即白
3. **AI太弱不如自己做**：新手经理的陷阱——亲力亲为是加法，好的管理是乘法
4. **AI太快质量无法保证**：引入层级结构，从M1升级到M2，用制度和自动化来保证质量

**关键升级路径**：
```
IC (Individual Contributor)
    ↓ 开始用AI
M1 (一线经理) - 管理几个AI
    ↓ AI数量增加
M2 (高级经理) - 管理的不是产出，而是工作流、制度、自动化
```

**这篇文章要回答的问题**：当你升级到M2，管理十几个AI agent的时候，团队结构应该是什么样的？

## 我们的补充角度：从愿景到路线图

黄东旭描绘了AI原生开发的愿景，我们要补充的是：**具体怎么落地？**

### 💡 黄东旭提出的问题，我们来回答

**黄东旭的愿景**：头狼+狼群，one-man army，强解耦并行

**黄东旭留下的问题**：
1. 头狼之间如何协作？（他说"同一片领地容不下两匹头狼"）
2. 质量如何保证？（他说"90%时间在测试验收"）
3. 瓶颈如何突破？（他说"唯一瓶颈是人工code review和运维"）

**我们的补充**：
- 铁三角：三个owner各自负责，不需要都是"头狼"
- 质量链：把质控分散到每个环节，不是集中在最后
- 自动化编排：/conduct + subagents解决瓶颈

### 💡 "头狼"模式的风险

**Bus Factor问题**：
> "When the lone wolf developer, for whatever reason, no longer works at the organization, you end up in a scramble to find replacements. Because the lone wolf developer never bothered to document what they did or shared their code with others."

来源: [DistantJob - Lone Wolf Problem](https://distantjob.com/blog/bringing-the-lone-wolf-into-the-pack/)

**企业级风险**：
> Java creator James Gosling: "vibe coding is not ready for the enterprise because in the enterprise, [software] has to work every fucking time."

来源: [The New Stack - Vibe Coding Fails Enterprise](https://thenewstack.io/vibe-coding-fails-enterprise-reality-check/)

**安全漏洞**：
- 45%的AI生成应用包含可利用的OWASP漏洞
- 205,000个幻觉包被发现（slopsquatting攻击面）

来源: [Veracode 2025研究](https://www.infoworld.com/article/4116937/output-from-vibe-coding-tools-prone-to-critical-security-flaws-study-finds-2.html)

### 💡 Nick访谈的反例证据：多层质量控制

Nick的模式恰恰不是"头狼"模式，而是**多层质量控制 + 分布式ownership**：

#### 质量控制点1：PRD由Product Owner负责

> "Product Owner是要为这个产品负责的...产品负责人在Scrum里是一个最重要的角色，他能对这个产品的所有事情拍板。"

> "用户故事只描述原始需求和一些验收标准。"

**PO owns PRD和用户故事的质量**——这不是AI能替代的。

#### 质量控制点2：Nick亲自review详细设计（Story Spec）

> "我现在review最多的就两件事：一个是Epic里的用户故事，另一个是每个用户故事对应的详细设计。"

> "你review完他那个详细设计文档，觉得验收标准、任务、子任务、代码分层那些都OK之后，就直接走一个dev_story的工作流程。"

**关键**：Nick不是让AI写完就跑，而是**每个Story的详细设计都要人工review**。

#### 质量控制点3：AI对抗式Code Review + Nick审核报告

> "code_review完之后我要看他描述的那些问题是不是对的。我看过他找出的问题，80%是正确的，但有时候不知道是因为上下文还是什么原因，他发现的某些问题其实已经实现了，或者是不正确的，你要纠正他。"

> "所以那个code_review的结果应该看一下...code_review的结果应该都是三到五个左右，也不算太多，看一下也很快。"

**Nick不看代码，但看Code Review的报告**——这是关键的质量控制。

#### 质量控制点4：测试用例的验收

> "AI在实现过程中会自动帮你添加很多单元测试，但说实话，这些单元测试在微观层面基本都能通过...但AI不擅长的是集成测试、端到端测试。"

> "我在完成大目标前，我一定会先和AI一起做一个方便的集成测试框架，并提前准备好测试的基础设施。"

马驰补充：
> "端到端测试，从业务流程的端到端测试，就是为了保证，如果从业务上我端到端能够实现，至少我的happy path能够跑通，我就可以上线了。"

### 💡 铁三角之外：支撑角色

Nick的铁三角（PO + UX + Tech Owner）只是前台，背后还需要支撑角色：

#### 支撑角色1：Platform Engineer（平台工程师）

**职责**：构建CI/CD pipeline，让AI可以调用

> 原文作者提到："唯一的瓶颈是人工的code review和一些无法被自动化的线上运维工作（我觉得也很快了）"

**关键**：如果CI/CD不是AI-ready的，AI写完代码还要人工部署，效率大打折扣。

Platform Engineer的产出：
- AI可调用的测试pipeline
- 自动化部署流程
- 监控和告警基础设施
- 一键回滚能力

**Nick的实践**：
> "我一般自己还会去做一下手工验收，因为他那个单元测试、集成测试、E2E测试都跑过了。"

这说明他已经有了自动化测试基础设施。

#### 支撑角色2：Architect（架构师）

**职责**：拆分微服务边界，定义系统架构

马驰的实践：
> "如果review不过来，那可能在更高层次上，比如Epic就要拆分好了，分为两个项目。"
> "我现在做架构图，把它分成多个微服务，每个微服务负责不同的东西。这样的话，一个大项目最终落地的时候也就是一个微服务而已。"

Nick的观点：
> "如果你是微服务的话，可能就要参考DDD，把那个领域划分清楚，才会比较少那些重合的情况。"

原文作者也承认：
> "我发现当前的Coding Agent在面对单一模块复杂度超过大约5万行代码之后，就开始很难在1-shot里把问题一次性解决掉"
> "Agent通常不会主动去做项目结构和模块边界的治理"

**结论**：AI不会主动拆分架构，需要人类架构师预先规划边界。

### 💡 铁三角的真正构成

原来以为是：Product Owner + UX Owner + Tech Owner

**修正**：Product Owner + Quality Controller + Tech Owner

#### 为什么是Quality Controller而不是UX Owner？

1. **UX不是每个项目都需要**：后端API、CLI工具、基础设施项目都不需要UX
2. **Quality Controller是AI原生开发的关键瓶颈**

原文作者的关键洞察：
> "我90%的时间和精力都花在了这个阶段：也就是如何评估AI的工作成果"
> "There's a test, there's a feature，你只要知道如何评估和测试你要的东西，AI就一定能把东西给你做出来。"
> "我在完成大目标前，我一定会先和AI一起做一个方便的**集成测试框架**，并提前准备好测试的基础设施"

**这正是Quality Controller的工作！**

#### Quality Controller的职责

1. **构建集成测试框架**（原文作者的做法）
2. **定义验收标准**（Nick强调的）
3. **端到端测试**（马驰的痛点）
4. **对抗式测试**（发现AI看不到的问题）

马驰的痛点恰恰是缺乏Quality Controller：
> "我现在想要引入一个角色叫做QC（质量控制）。因为我的场景，第一，我在银行项目里有很多测试是没办法自动化的。"
> "我希望有一个人跟我对抗式测试，我写出来觉得挺好的，他一定给我找茬，说这个不行、那个不行，这样效率其实是更高的。"
> "我自己找我自己的茬总是会有点懒惰，'也差不多了，挺好的了'。"

**关键**：AI写代码时不会考虑边界条件，需要人来定义"什么是对的"。

### 💡 原文作者的工作流 = BMAD/Conduct的理念

原文作者描述的工作流：
> "首先让 GPT-5.2 在 Codex 下生成多个功能的设计文档...然后在第二阶段，依然用 Codex 根据这些需求文档一个一个去实现功能...在接近完成的时候，在代码通过测试并准备提交之前停下，把当前的工作区交给另一个 ClaudeCode 或 OpenCode，在不提供上下文的情况下，让 ClaudeCode 来 Review 当前还未提交的代码...然后再把这些建议发回给 Codex，让 Codex 来评论这些建议...直到双方都觉得代码已经写得很不错了，再提交到 Git 上"

**这正是多Agent协作的模式！**

#### 原文作者的工作流分解：

```
1. GPT-5.2/Codex → 设计文档
2. Codex → 实现功能
3. Claude/OpenCode → Review（无上下文，对抗式）
4. Codex → 根据建议修改
5. Claude → 再次Review
6. 循环直到双方满意 → Git提交
```

#### 对应到BMAD/Conduct的角色：

| 原文工作流步骤 | BMAD角色 | Conduct中的subagent/skill |
|--------------|----------|--------------------------|
| 生成设计文档 | Architect/SM | /architect, /story |
| 实现功能 | Developer | coder subagent |
| 对抗式Review | QA/Reviewer | reviewer subagent |
| 修改代码 | Developer | coder subagent |
| 测试通过 | Tester | tester subagent |
| Git提交 | - | /qc skill |

#### 我们的推荐：用Conduct编排整个流程

原文作者**手动**编排多个Agent，但这可以自动化：

```
/conduct 或 自定义的workflow skill
    │
    ├── 1. /story → 生成详细设计
    │
    ├── 2. tester subagent → 写测试（TDD红阶段）
    │
    ├── 3. coder subagent → 实现代码（TDD绿阶段）
    │
    ├── 4. reviewer subagent → 对抗式Review
    │
    ├── 5. coder → 根据review修改
    │
    ├── 6. /qc → 质量检查 + Git提交
    │
    └── 7. deployer → 部署（如果需要）
```

**关键差异**：
- 原文作者：手动切换Agent，手动传递上下文
- BMAD/Conduct：自动编排，skill之间自动传递

#### 为什么这更好？

1. **可重复**：每个Story走同样的流程
2. **不遗漏**：不会忘记某个步骤
3. **可审计**：每步都有记录
4. **节省时间**：不用手动切换

Nick的实践（BMAD）：
> "有了用户故事，下一步就是写详细设计文档...你review完他那个详细设计文档...就直接走一个dev_story的工作流程。他就会按照你的那个详细设计全部实现。实现完之后有一个可选流程，就是做自动化测试...开发完并做完单元测试之后，还可以走一个叫code_review的流程。"

**Nick和原文作者做的事情本质相同，只是Nick用BMAD把它固化成了流程。**

### 💡 原文作者自己承认的瓶颈

> "而且唯一的瓶颈是人工的code review和一些**无法被自动化的线上运维工作**（我觉得也很快了）"

**但这两个瓶颈都可以解决！**

#### 瓶颈1：人工code review → reviewer subagent

原文作者自己已经在用了：
> "让ClaudeCode来Review当前还未提交的代码，根据设计提出修改建议"

Nick的实践（BMAD的code_review流程）：
> "code_review流程比v4版本强很多，它是一个对抗式code review。它会根据你的详细设计，只要有任何一点不符合，他都找出来告诉你。"

**关键**：不是让AI替代人工review，而是让AI做第一轮筛查，人只看report。

- 更新配置

## 核心框架：铁三角

**铁三角不只是团队结构，它是一个解决问题的框架**

黄东旭提出的问题，铁三角如何结构性地解决：

| 黄东旭的问题 | 铁三角的解决方式 |
|-------------|-----------------|
| 90%时间在测试验收 | **Quality Controller**专职负责，构建集成测试框架 |
| 人工code review是瓶颈 | **Tech Owner**只看report，reviewer subagent做初筛 |
| 运维无法自动化 | **Platform Engineer**（支撑角色）构建AI可调用的基础设施 |
| 领地容不下两匹头狼 | 三个Owner各有领地：产品/质量/技术，不冲突 |
| token消耗二八定律 | 结构化流程（/conduct）让普通人也能高效 |

**铁三角的本质**：把"头狼"的工作分解成三个可分工的ownership领域

```
头狼一个人做的事：
  - 定义做什么 ──────→ Product Owner
  - 定义什么是对的 ──→ Quality Controller
  - 定义怎么做 ──────→ Tech Owner
```

**为什么铁三角比头狼更好？**

1. **Bus factor降低**：三个人各own一块，一个人离开不会全崩
2. **专业分工**：QC专注测试框架，Tech Owner专注架构，PO专注业务
3. **可扩展**：大项目加Architect和Platform Engineer，小项目可兼任
4. **瓶颈分散**：不是一个人承担所有决策压力

      │                        │                        │

**核心洞察**：

| 路线 | 改变什么 | 问题 |
|------|----------|------|
| 精英路线 | 换人 | 人才稀缺，"做不到就淘汰"太残酷 |
| 温和路线 | 换工具 | 不动组织结构，效果有限 |
| **铁三角** | **换结构** | 需要组织变革，但人才可培养 |

### 💡 关键漏洞：头狼/产品架构师在哪里？

**黄东旭的"头狼" = 许式伟的"产品架构师"**

马工在群里的追问：
> "马工: @许式伟 这个试验有最新进展吗？我上次问你，你还是你公司唯一一个产品架构师"
>
> "马工: 产品架构师这个岗位的挑战太大了，这群里也就@胥克谦 @王欢.ai 接近这个定位"
>
> "马工: 这个产品架构师职责倒是挺清晰的，一个人端到端负责，不过它需要什么技能？**不可能是传统的产品经理，架构师和lead engineer的技能的简单合集吧**"

许式伟的回答：
> "许式伟: 不是 是全员啊"
> "许式伟: 我的逻辑比较简单 如果一定时间内做不到端到端的就淘汰 换其他能够做到端到端的来"

**问题暴露了**：
- 马工问"你还是你公司唯一一个产品架构师"，说明这个角色的稀缺性是共识
- 全群几百人，马工只能想到2个人"接近这个定位"
- 许式伟的解决方案：**筛选淘汰**，而不是培养

**这正是精英路线的致命问题**：
- 产品架构师/头狼需要：产品经理 + 架构师 + Lead Engineer的能力
- 这种人本来就稀缺
- AI时代不是凭空创造了这种人才，只是让已有的顶尖人才更强
- 大多数公司**找不到这样的人**

### 💡 铁三角的价值：把"不可能的人"拆成"三个可能的人"

**产品架构师需要的技能**（马工的分析）：
- 产品经理技能：理解业务、定义需求、做取舍
- 架构师技能：系统设计、模块拆分、技术选型
- Lead Engineer技能：代码质量、技术债务、团队协作

**铁三角的拆分**：
```
产品架构师（一个人做所有事）
        │
        │ 拆分
        ▼
┌───────────────┬───────────────┬───────────────┐
│ Product Owner │    Quality    │  Tech Owner   │
│               │   Controller  │               │
├───────────────┼───────────────┼───────────────┤
│ 产品经理技能   │ QA/测试技能    │ 架构师技能    │
│ + 业务判断力   │ + 验收标准    │ + Lead Eng技能│
└───────────────┴───────────────┴───────────────┘
```

**为什么这样拆分可行？**

1. **每个角色的技能是存在的**：
   - 好的Product Owner很多（传统PM转型）
   - 好的QA/测试很多（加上AI工具更强）
   - 好的Tech Lead很多（现有Senior Engineer）

2. **组合比寻找更容易**：
   - 找一个"产品架构师" → 极难
   - 找三个各自领域的专家 → 可行

3. **培养比筛选更可持续**：
   - 许式伟："做不到就淘汰" → 人从哪来？
   - 铁三角：明确每个角色的职责 → 可以针对性培养

**这就是铁三角的核心价值**：
> 不是说"产品架构师"不好，而是说大多数公司找不到。
> 铁三角把这个"不可能的超级个体"拆成了"三个可能的专业角色"。

**为什么铁三角是中间路线？**

1. **不像精英路线那么难**：
   - 许式伟："做不到端到端就淘汰" → 人才在哪里？
   - 黄东旭："顶级工程师+$200模型" → 谁付得起？
   - 铁三角：三个人分工，每个角色的技能是**可学习的**

2. **不像温和路线那么弱**：
   - 杨正武：定制skills给实施组用 → 没有触及ownership问题
   - 问题：谁为PRD质量负责？谁为测试覆盖率负责？谁为架构决策负责？
   - 铁三角：明确**三个Owner**，每人对自己领域的结果负责

3. **铁三角要求组织变革，但值得**：
   - 不是简单地"培训员工用AI工具"
   - 而是重新定义**谁负责什么**
   - 这才是AI时代团队的根本问题

**文章的核心论点**：

> AI时代的团队问题，不是"人怎么用AI工具"，而是"谁为什么结果负责"。
>
> 精英路线要求超级个体一人负责一切——人才难找。
> 温和路线只给工具不动结构——效果有限。
> 铁三角把"一切"分解成三个ownership领域——可学习、可分工、可追责。
