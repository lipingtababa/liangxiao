# Outline: BMAD，AI时代的敏捷还魂术

## 模板：驳斥（Debunking）— 永动机模式

## 核心论点
人与 AI 之间确实存在对齐问题，但问题的性质和人与人之间完全不同。BMAD 把 Scrum 的人际对齐流程搬给人与 AI 的协作，用治政治病的药方去治统计病，方向搞反了。

## 标题候选
- BMAD，AI时代的敏捷还魂术
- BMAD搞反了方向

## 结构

### 1. 开场：群聊引入
- 最近群里爆发了一场关于 BMAD 的激烈辩论
- 支持者说它是目前最高水平的 AI 软件工程框架
- 反对者说它又臭又长
- 我承认 BMAD 很好，但我就是不喜欢它，就像我承认刘亦菲五官精致，但我就不喜欢她
- 但"不喜欢"不是论证。到底哪里不对？

### 2. Brooks 的外科手术团队：一个被验证了四十年的模型
- 《人月神话》：概念完整性是系统设计最重要的属性
- 外科手术团队模型：主刀医生负责概念完整性和关键决策，支援团队在各自专业范围内工作
- 关键洞察：协作效率不来自每个人都懂全局，而来自角色边界清晰、接口协议明确
- 主刀不需要懂麻醉，麻醉师不需要会开刀

### 3. Scrum 解决的是人与人的问题（理论基础）
- 传统 Scrum 三个角色存在的原因：信息不对称、利益不一致、沟通带宽有限
- Sprint Planning、Daily Standup、Sprint Review 本质上都是人际对齐仪式
- 这些仪式有成本，但在人与人协作时值得，因为不对齐的代价更高

### 4. BMAD 的根本错误：用抗生素治骨折
- BMAD 做了什么？把 Scrum 角色变成 AI agent，让人类按 Scrum 流程逐一对齐
- 人与人的对齐问题是政治性的：信息不对称、利益冲突、沟通瓶颈。Scrum 仪式是谈判桌。
- 人与 AI 的对齐问题是统计性的：LLM 倾向于给一个"看起来完成了"的答案快速收工、为了节省 token 走捷径、讨好用户说"好的没问题"然后糊弄、在长 context 里丢三落四
- AI 不是"不想做好"，而是"不知道什么叫做好"。这不是动机问题，是校准问题。
- 对人的正确手段：Scrum 仪式、Sprint Review、角色分工（解决政治病）
- 对 AI 的正确手段：高密度自动化测试、精准 context 工程、迭代式反馈（解决统计病）
- BMAD 用 Sprint Planning 去矫正 LLM 的统计偏差，就像用抗生素治骨折。感染是真的，但骨折需要正骨，不需要消炎。

### 5. 方法论演化史：每一步都在消除对齐成本
- 瀑布 → 敏捷：砍文档签字流程
- 敏捷 → 精益：砍 Sprint 边界僵硬仪式
- 精益 → DevOps：砍开发和运维之间的组织墙
- DevOps → 平台工程：砍开发者和基础设施之间的工单
- 每一步的模式：识别因人的局限性产生的对齐成本 → 用技术消除局限性 → 消除成本
- BMAD 逆转了这个方向：把人际协作的仪式成本重新引入人机协作，用政治手段解决技术问题

### 6. 正确的方向："裁剪悖论"怎么破
- 胥克谦的洞察：有能力剪裁流程的人不需要框架，需要框架的人没有剪裁能力
- 正确的方向是 Brooks 的外科手术模型在 AI 时代的推论：人类是主刀，AI 是整个支援团队的集合体
- 固定人类的工作流，让 AI 在每个环节自适应角色
- 人类不需要分别和 PM agent、architect agent、UX agent 开会
- 具体的做法参考：胥克谦的四步固定流程 + 角色自适应

### 7. 结尾
- BMAD 是个方法宝库，里面有大量有价值的素材
- 但它本身的设计方向是反的：让人适配流程，而不是让流程适配人
- 用治政治病的药方去治统计病，把仪式成本转嫁给用户
