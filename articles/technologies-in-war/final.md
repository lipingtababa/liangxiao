# 向四野炮兵学习AI应用策略

AI Coding先行者都在讨论同一个问题：pilot做完了，接下来怎么推组织层面的规模化？讨论来讨论去，常见的结论无非是"缺预算"、"缺executive support"、"需要更多时间探索"。似乎所有的问题都指向外部制约。

80年前，朱瑞在东北也遇到过类似的挑战。1945年9月，他空手到了东北。他有1000多名延安炮校的师生，有一个理想的职务——四野炮兵司令，但没有钱、没有人、没有时间。要建立一支系统化的炮兵部队，需要庞大的资源和组织。

他怎么做的？他直接立下军令状："给我钱给我人，三个月内，我给你四个炮兵团上前线打仗。"

这不是"试试看"。这是具体的数字（四个团）、明确的时间（三个月）、可验证的结果（上前线打仗）。而且他真的做到了。1947年12月的彰武战役，朱瑞集中火炮对敌阵地进行压制，歼敌600余人。1948年3月的四平战役，四野炮兵以每分钟500发的火力强度轰击敌阵，23小时攻克四平。到1948年底，四野已拥有9000门火炮，炮兵部队成为东北战场上最有威力的力量。三年后，从东北打到海南，炮兵部队在三大战役中成为决定性的力量。

朱瑞的故事告诉我们，很多"问题"其实是伪问题。真正的问题从来不是缺什么，而是敢不敢做什么。

在过去一年里，我看到很多AI Coding的先行者陷入了这样的困局。他们手里有Claude Code、Cursor这样的工具，也有初步的成果，但在推向组织层面的规模化时，就开始退缩。不是因为工具不够好，而是因为没有人愿意打包票。

---

## "缺乏预算/executive support"的伪问题

很多AI Coding lead说："老板不给预算，没有executive support。"听起来合理。但这是把问题外部化，就像说"缺火炮"一样，实际上反映的是敢不敢向上面要。

朱瑞向东北局要了什么？500多名干部、完整的炮兵司令部、大笔的经费。但关键不在于他要的数字有多大，而在于他怎么要的。他没有说"试试看能不能建立炮兵部队"，而是立下军令状："三个月内，装备训练出四个炮兵团开赴前线打仗。做不到，我下台。"

具体数字。明确时间。可验证结果。个人风险。这个组合改变了一切。东北局看到的不是"我们来探索一下"，而是"他敢打这个包票"。

AI Coding lead的常见做法是什么？"我们试试Claude Code或Cursor，看能不能提升生产力。需要一笔预算用于license和培训。"对比一下朱瑞的做法："给我10个最强工程师、Claude Code企业版（或等价的专业AI编码工具）、3个月时间。我保证：第一个月交付2个features证明3-5倍的生产力，第二个月train 10个人，第三个月交付5个features。做不到，我下台。"

听起来很不一样。为什么？因为前一种是在推销一个理想，后一种是在承诺一个结果。高层最看重的不是"试试"的精神，而是"敢赌"的勇气。朱瑞不是在问东北局"能不能给我资源"，他是在对东北局说"我拿结果换资源"。

这牵涉到一个双向承诺的关系。朱瑞敢打包票+交付，东北局批资源+保护。在AI Coding的语境里，这是CTO批资源和保护，AI Coding Lead打包票和交付。不是单向的"赋能"，而是双向的承诺。承诺的存在改变了游戏规则。有了承诺，CTO就有底气问董事会要预算，董事会就敢批。没有承诺，一切都是"我们在摸索"。

## "全员培训"的伪问题

常见的做法是什么？给1000个开发者买license，办培训班，期待自然提升。六个月后，使用率15%，效果不明显，然后得出结论"开发者还需要更多培训"。问题被无限延后。

朱瑞是怎么处理的？他没有培训所有步兵军官使用火炮。相反，他组建了专业的炮兵单位——从连、营、团，一直到司令部。他派遣500多名干部到各军区，这些干部不是去教所有步兵怎么用炮，而是去建立专业的炮兵组织。

为什么？因为炮兵指挥官和步兵指挥官需要的完全是两套不同的技能和思维。强行让步兵官学炮兵，效率太低，浪费资源。更高效的做法是，集中精力把一小批人培养成专业的炮兵，然后让他们与步兵配合。

转到AI Coding，这意味着什么？不是1000个开发者里每个人都学会，而是10-20个专职AI Coding Specialist。这些人不是"有空就试试AI工具"，而是专业角色。他们接受两周的集中培训（bootcamp级别），变成真正的AI Coding specialist，平时embed到产品团队里，关键时刻集中使用。

集中使用什么时候发生？朱瑞有一个原则："广泛的普遍的发展与适当的集中使用"。在锦州总攻的时候，他集中了591到900门火炮，在31小时内歼灭了10万国军。在天津战役，他集中538门火炮，29小时歼灭13万。这些数字为什么能造成决定性的结果？因为它们都是集中投入关键战场的。

炮兵真的有效吗？最有说服力的评价来自于朱德。在《学习毛泽东军事思想》中，他总结说：

> "能消灭了敌人主力，炮是起了主要作用的。"

更进一步，毛泽东在《将革命进行到底》中指出：

> "自从人民解放军形成了超过国民党军的炮兵，国民党的防御体系就显得渺小了。"

这不是朱瑞一个人的想象，而是三年实战中被验证的事实，以至于朱德、毛泽东这样的军事领导人都不得不承认炮兵的决定性作用。这就是为什么朱瑞敢打包票、敢要资源——因为炮兵已经被证明了，不只是被朱瑞证明，而是被整个战争进程和最高领导人都证明了。

在AI Coding的语境里也是一样。当公司要推出一个重要产品发布的时候，把所有10个AI Coding specialist集中起来，部署到一个团队里，6周内形成压倒性的优势。不是"AI everywhere"，而是"AI where it matters"。关键是，你先要能证明它有效。

## "需要更多时间探索"的伪问题

常见的说法："我们需要一年时间来探索AI Coding的最佳实践。"或者"我们要慢慢建立AI Coding文化。"6个月后，还在摸索中。问题变成"还需要继续探索"。

看朱瑞的速度。1945年9月到东北，空手一身。1946年5月，他搜集了700多门火炮和50万发炮弹——8个月。1946年10月，炮兵司令部成立，100多个炮兵连——13个月。1948年底，9000门火炮——3年。

每个阶段都有明确的deliverable。不是"持续建设中"，不是"还在摸索"，而是"已经交付"。

怎么做到的？3个月一个cycle的节奏。朱瑞的cycle是：8个月装备 → 4个月组织 → 3个月部队 → 12个月实战 → 3个月战果。每个cycle他用成果换下一轮资源。没有资源的时候，他问的不是"怎么继续"，而是"我已经交付了什么，接下来需要什么"。

在AI Coding的scale-up里，这个3个月的cycle依然适用：Month 1-3 是pilot（交付pilot的成果），Month 4-6 是CoE的建立（交付training material和best practices），Month 7-12 是规模化（交付数字）。用成果换资源。不是"我们在探索"，而是"我们已经证明了3倍生产力，接下来需要10个新specialist"。

朱瑞自己说过一句话："历史机时，稍纵即逝，不抓是不对的。"他说这句话的背景是什么？1945年到1946年是唯一的机会大规模搜集装备。那时国民党的仓库还没来得及转移，日本的军事装备还在东北。错过这个窗口，就再也搜不到这么多火炮。

2026年的AI Coding有类似的窗口期吗？我认为有。早期行动者（从2024年或2025年起步）已经领先12到18个月。2027年，AI编程会成为基本功，就像今天所有开发者都会用GitHub一样。到那时，早期行动者建立的优势消失了，因为成了行业标准。现在开始的还有6到12个月的窗口，但这个窗口在快速关闭。

## 从朱瑞学到的5个原则

原则1：**敢打包票，用具体数字。** 不是"试试看"，不是"我们在摸索"，而是具体的数字 + 明确的时间 + 可验证的milestone + 个人风险。朱瑞对东北局说的就是这样：不是"试试能不能组建炮兵"，而是"三个月、四个团、上前线"。AI Coding Lead对CTO也应该这样说。有了这个承诺框架，CTO才敢向董事会要预算。

原则2：**敢要资源。** 不是"有多少用多少"的被动态度，而是"要办成事需要多少"的主动态度。朱瑞要500个干部、司令部、经费——他算出来这些才能做成事。AI Coding Lead要10个specialist、license、3个月时间——这也是算出来的。要得越具体、理由越充分，越容易被批。

原则3：**高层承诺 + 技术交付的合作。** 这是一个双向关系，不是单向的。CTO批资源+保护，Lead打包票+交付。朱瑞能放心要资源，是因为林彪承诺了支持。反过来，林彪敢批这些资源，是因为朱瑞承诺了结果。这种双向承诺改变了一切。

原则4：**建专业团队 + 集中使用。** 不是全员培训1000个开发者，而是10-20个专职AI Coding specialist。他们不是"有空就用用Claude Code"，而是这是他们的专业工作。平时分散embed到各个team维持现状，关键时刻集中到一个项目形成压倒性优势。这就是朱瑞说的"广泛的普遍的发展与适当的集中使用"。

原则5：**快速迭代验证，3个月一个cycle。** 每个cycle有明确的deliverable，不是"我们在持续探索"。用成果换资源。Month 1-3交付成果，Month 4-6交付下一个成果。问题不问"需要多久"，只问"我们已经做到什么"。这样既避免了无限探索的黑洞，也给了自己争取下一轮资源的依据。

这五个原则都不是技术原则，都是组织和领导力的原则。朱瑞成功的秘密不在于他比别人更懂火炮，而在于他知道怎么向上要资源、怎么向下交付结果、怎么在组织内部建立信心和期待。

## 2026年的现实：你已经晚了

对于AI Coding的规模化，有三个tier。

**Tier 1：先行者** 从2025就开始的，已经有训练好的专家，已经验证了5到10倍的生产力。他们的优势已经锁定。

**Tier 2：快速跟随者** 现在开始，还有6个月的紧急sprint，还能建立一个可观的优势。但这个窗口在快速关闭。

**Tier 3：落后者** 从2027年开始的，AI编程已经成为基本功，没什么竞争优势。

朱瑞的紧迫感在2026年依然适用。他说"资材遍地，先入为主，历史机时，稍纵即逝，不抓是不对的"。现在是什么情况？Claude Code ready，Cursor ready，GitHub Copilot ready，各种工具ready。窗口还有6到12个月。

如果你不抓，两年后你的竞争者已经建立了什么？他们训练了500个specialist，验证了稳定的10倍生产力，建立了自己的最佳实践库。他们知道在什么项目上用AI能产生最大杠杆，知道怎样组织团队让AI coder和人类开发者配合无缝。你要再想追，就太晚了。那时候，这已经不是"先发优势"的问题，而是完全不在同一个竞争维度。

这个说法也许太绝对。但朱瑞的案例告诉我们，有些窗口错过了就真的错过了。他在1945年的那个决定改变了接下来三年的战争走势。你在2026年的这个决定，也许会改变你的组织接下来三年的竞争力。区别可能就在敢不敢打包票。
